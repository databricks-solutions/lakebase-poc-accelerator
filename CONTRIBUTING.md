# Contributing to the Lakebase POC Accelerator

Thanks for helping improve the accelerator! This guide covers **local setup, the dev
workflow, project layout, and deployment**. For what the app *does* (end-user
capabilities), see the [README](./README.md).

The app is built on the **[apx](https://github.com/databricks-solutions/apx)** toolkit:
FastAPI + Pydantic backend, React + TypeScript + shadcn/ui frontend, `uv` for Python and
`bun` for JS.

---

## 🧰 Prerequisites

| Tool | Why | Install |
|------|-----|---------|
| **Python ≥ 3.11** | Backend runtime (see `.python-version` → `3.11`) | [python.org](https://www.python.org/) |
| **[uv](https://docs.astral.sh/uv/)** | Python dependency & venv manager (not pip) | `curl -LsSf https://astral.sh/uv/install.sh \| sh` |
| **[bun](https://bun.sh/)** | JS package manager & runtime (not npm) | `curl -fsSL https://bun.sh/install \| bash` |
| **[apx](https://github.com/databricks-solutions/apx)** | Dev orchestration (backend + frontend + OpenAPI watcher) | See the apx repo |
| **[Databricks CLI](https://docs.databricks.com/dev-tools/cli/)** | Auth to a workspace + bundle deploy | `brew install databricks` |

You also need access to a **Databricks workspace** with **Lakebase** enabled, and a
configured CLI auth profile (`databricks auth profiles`). The app authenticates to
Lakebase via your Databricks identity — no stored secrets. `.env` files are gitignored.

## ⚙️ Local setup

```bash
# 1. Clone
git clone <repo-url> && cd lakebase-poc-accelerator

# 2. Install dependencies
uv sync            # Python deps into .venv
bun install        # JS deps

# 3. Authenticate to your workspace (dev token flow)
databricks auth login --host https://<your-workspace>.databricks.com
```

For local runs, the app uses the **OAuth token** auth method (see the README's
Authentication section) — you paste an endpoint host, Postgres user, and a short-lived
token from Lakebase Connect. Tokens expire after ~1 hour.

## 🔁 Development workflow

`apx dev` runs the backend, the Vite frontend, **and** the OpenAPI watcher (which
regenerates the frontend API client) together in the background.

```bash
apx dev start        # start everything (detached)
apx dev logs -f      # tail logs
apx dev status       # health + local URL (e.g. http://localhost:9005)
apx dev stop         # stop everything
apx dev check        # typecheck + lint (TypeScript & Python) + health
```

Quick frontend typecheck without apx:

```bash
npx tsc --noEmit -p tsconfig.json
```

### ⚠️ The generated API client — do not hand-edit

`src/lakebase_accelerator/ui/lib/api.ts` is **generated by [Orval](https://orval.dev/)**
from the backend's OpenAPI schema. **Never edit its generated sections by hand.** To
change the API:

1. Edit the Pydantic models / routes in the backend (`backend/models.py`, `backend/routers/*.py`).
2. The apx OpenAPI watcher (running under `apx dev`) regenerates `api.ts` automatically,
   giving you typed react-query hooks (`useRunPsycopgTest`, `useSubmitPgbenchJob`, …).

## 🗂️ Project structure

```
pyproject.toml                       # Python project + [tool.apx] config (app-slug, entrypoint, api-prefix=/api)
databricks.yml / app.yml             # Databricks Asset Bundle + Apps config
package.json / tsconfig.json         # Frontend (bun) config, at repo root
docs/images/                         # README screenshots
src/lakebase_accelerator/
  backend/
    app.py                           # FastAPI entrypoint (…backend.app:app)
    router.py                        # Aggregates routers under the /api prefix
    routers/                         # testing, deployment, optimize, history, lakebase
    services/                        # Business logic (auth, connection, pgbench_job, optimize, …)
    core/                            # App wiring, logging, lakebase URL helpers
    deps.py                          # FastAPI deps (EffectiveClient = OBO WorkspaceClient)
    models.py                        # Shared Pydantic models
    resources/                       # Bundled job assets (pgbench notebook + init script)
  ui/
    main.tsx                         # React entry
    routes/                          # TanStack Router pages: index, deployment, testing, cost, best-practices, docs
    components/                      # Feature + shadcn/ui components
    lib/api.ts                       # Orval-GENERATED API client (do not hand-edit)
    lib/history.ts                   # localStorage run-history store
```

### Architecture at a glance

- **Backend** — FastAPI; routes split across `routers/*.py`, each via `core.create_router()`,
  aggregated in `router.py`, served under `/api`.
- **DB driver** — **psycopg 3** (`import psycopg`); SQLAlchemy uses `postgresql+psycopg://`.
  No psycopg2 / asyncpg.
- **Concurrency runner** (`services/connection.py`) — SQLAlchemy `QueuePool` over psycopg3,
  with a dedicated `ThreadPoolExecutor` sized to the concurrency level.
- **pgbench** (`services/pgbench_job.py`) — runs as a Databricks Job on a single-node
  cluster. **Hybrid identity:** the job is created/owned/run by the app **service
  principal** (the Jobs API needs scopes the OBO user token lacks), while Lakebase
  credentials are resolved as the **user** (OBO) and passed into the job, so the benchmark
  connects to Postgres as the user. Run-history follows the same pattern (SP owns the
  table; `created_by` captures the user).
- **Frontend** — React 19 + Vite + TanStack Router (file-based routes) + TanStack Query;
  shadcn/ui (Radix + Tailwind), `lucide-react`, `sonner`.

## ✅ Coding conventions

**Python** — type hints on all signatures; snake_case funcs/vars, PascalCase classes.
Pydantic `BaseModel` for API models, `dataclass` for internal config. `async`/`await`
for I/O; offload blocking psycopg work to a thread executor. **Parameterized queries
only** — never string-format user SQL; identifiers that must be interpolated are
validated against `^[a-zA-Z_][a-zA-Z0-9_]*$` first.

**TypeScript / React** — functional components; camelCase vars, `handle*`/`on*` for
handlers; shadcn/ui + Tailwind with `cn()` from `lib/utils.ts`. Don't hand-edit generated
parts of `lib/api.ts`.

Run `apx dev check` before opening a PR.

## 🚢 Building & deploying

```bash
# Production build
apx build

# Validate + deploy the bundle to a Databricks workspace
databricks bundle validate -p <profile> -t <target>
databricks bundle deploy   -p <profile> -t <target>

# Deploy/start the app so it picks up the new code
databricks bundle run lakebase-accelerator-app -p <profile> -t <target>
```

Targets are defined in `databricks.yml` (e.g. `dev` is the default; `qa` is bound to a
specific workspace host). Each target has separate Terraform state, so deployments don't
collide.

> **pgbench needs the app service principal to create clusters.** If a pgbench run fails
> with `PERMISSION_DENIED: You are not authorized to create clusters`, grant the app's
> service principal the *"Allow unrestricted cluster creation"* entitlement (Settings →
> Identity and access → Service principals), or select an existing cluster in the Testing
> UI so no new cluster is created. The app surfaces this suggestion in the failed run's
> error box.
